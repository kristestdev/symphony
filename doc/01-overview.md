# 01 总览：Symphony 是什么

## 一句话

**Symphony** 把「项目里要做的事」变成**隔离的、可重复的自动化运行**：从工单系统（当前规范为 **Linear**）拉取候选任务，为每个工单创建独立工作区，在区内启动 **Codex App-Server** 执行编码代理会话；团队通过管理工作流与工单状态来推进工作，而不是盯着每一次 Agent 对话。

详见仓库 [README.md](../README.md)。

## 解决的四类运维问题（SPEC 视角）

根据 [SPEC.md](../SPEC.md) 第 1 节，服务针对：

1. **可重复的长驻流程**：把「手工脚本跑 Agent」变成守护进程式编排。
2. **按工单隔离**：每个工单的命令只在对应工作区目录下执行，降低互相污染。
3. **策略版本化**：运行时策略与 Prompt 写在仓库的 `WORKFLOW.md` 中，与代码一起评审、回滚。
4. **可观测性**：结构化日志、可选仪表盘/API，便于并发多会话时排障。

## 明确边界（非常重要）

Symphony 在规范中的角色是：

| 是 | 不是 |
|----|------|
| 调度器 / 执行器 / 工单读取方 | 通用工作流引擎或分布式任务调度平台 |
| 按配置轮询、派发、重试、对账 | 规定「如何改 PR、如何评论工单」的业务逻辑（这些在 Prompt 与 Agent 工具里） |
| 可选提供观测与控制面（HTTP） | 必须依赖富 Web 控制台才能正确运行 |

**工单写入**（状态流转、评论、链 PR）通常由 **编码代理在会话内** 通过 MCP、`linear_graphql` 等工具完成，而不是由编排器内置「写 Linear API」的业务层完成（规范强调「编排器以读为主」；Elixir 实现里 `Tracker` 适配器可能提供少量写能力用于测试或扩展，见 [08-linear-tracker.md](08-linear-tracker.md)）。

一次「成功」的运行可以停在工单定义的交接态（例如 `Human Review`），而不必等于 Linear 的 `Done`。

## 目标与非目标（摘要）

摘自 [SPEC.md](../SPEC.md) 第 2 节：

**目标（节选）**

- 固定节拍轮询工单系统，有界并发派发。
- 单一编排器权威状态：派发、重试、对账。
- 确定性、可复用的按工单工作区；工单进入终态时停止运行并清理（策略见 SPEC）。
- 从瞬时故障恢复（指数退避等）。
- 从 `WORKFLOW.md` 加载运行时与 Prompt。
- 至少结构化日志；可选状态面。

**非目标（节选）**

- 多租户控制面、大而全的 Web UI（可有最小观测 UI）。
- 强制某一种沙箱/审批策略（由实现与部署文档说明）。

## 与「Harness engineering」的关系

README 建议目标代码库采用 [Harness engineering](https://openai.com/index/harness-engineering/)：Symphony 是在此之上的「**从管 Agent 到管工作**」的一步——工单与验收标准驱动，代理在隔离区内执行。
